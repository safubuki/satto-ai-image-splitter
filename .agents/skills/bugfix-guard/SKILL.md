# Bugfix Guard Skill — デグレ防止付きバグ修正・機能追加

## スキル読み込み通知

このスキルが読み込まれたら、必ず以下の通知をユーザーに表示してください：

> 🛡️ **Bugfix Guard スキルを読み込みました**  
> プロジェクト文脈を確認し、デグレ防止を意識して安全に修正します。

## When to Use

- コードの修正・機能追加を依頼されたとき
- 「〜を直して」「〜を追加して」「〜を修正して」と言われたとき
- 既存機能への影響を抑えて安全に変更したいとき
- 不具合の再発防止まで含めて対応したいとき

## 手順

### Step 1: プロジェクト文脈の確認（必須）

1. `*-overview` スキルを `.agents/skills/` と `.github/skills/` の両方から探索する
2. 候補が見つかったら、**現在のリポジトリに関連する内容か**を確認する
3. 関連している場合のみ、必要なリファレンスを読む（全件読込は不要）
4. 関連する overview が無い、または内容不一致の場合は以下を読む：
   - `README.md`
   - `package.json`（scripts と依存関係）
   - 今回変更対象の周辺コード

> 🛡️ 重要: 別プロジェクト由来の overview を無条件に適用しない。ファイル構成・技術スタック不一致なら参照対象から外す。

ユーザーへの報告フォーマット：

> 📋 **プロジェクト状況を確認しました**  
> - 参照した文脈: {overview名 または README/package.json}
> - 関連する既存パターン: {該当パターン}
> - 競合リスク: {あり/なし}
> - 注意点: {実装時の要点}

### Step 2: 影響範囲の特定

1. 変更対象ファイルを確定する
2. import 元/先を辿って関連箇所を洗い出す
3. 状態・データ・UI のどこに影響するかを分解する
4. 以下チェックリストを使って見落としを防ぐ

📄 **[references/degre-checklist.md](references/degre-checklist.md)**

### Step 3: 修正実装

以下の原則を守る：

| # | 原則 | 説明 |
|---|------|------|
| 1 | 既存挙動の尊重 | 無関係な挙動やUIを変更しない |
| 2 | 最小変更 | 影響範囲を必要最小限に抑える |
| 3 | データ整合性 | `box_2d` の形式・スケール（0-1000入力/0-1内部）を崩さない |
| 4 | 防御的実装 | 失敗点に `try-catch` / ガードを入れる |
| 5 | リソース管理 | `URL.createObjectURL` 追加時は解放を考慮する |
| 6 | 秘密情報保護 | APIキーをログやエラー文に出さない |
| 7 | 既存スタイル踏襲 | TypeScript/ESLint/命名規約を維持する |

### Step 4: 検証（必須）

`package.json` の scripts 存在を確認し、あるものだけ実行する。

1. `test:run` があれば実行
2. `lint` があれば実行（推奨）
3. `build` があれば実行（必須）
4. 手動スモークテスト（推奨）
   - 画像アップロード
   - 解析開始
   - 矩形編集
   - 切り抜き実行
   - ダウンロード

> ⚠️ 実行できない検証がある場合は「未実施理由」を完了報告に明記する。

### Step 5: 知見の反映

- 既存 overview がこのプロジェクトに関連している場合: 該当リファレンスへ追記する
- overview が無い/不一致の場合: 今回の学びを `bugfix-guard` 参照資料またはプロジェクトドキュメントに反映する
- 既存記述を破壊せず、追記ベースで更新する

### Step 6: 完了報告

以下フォーマットで報告する：

```markdown
## 対応サマリー

### 変更ファイル一覧
| ファイル | 変更内容 |
|---------|---------|
| {ファイルパス} | {変更内容の要約} |

### 検証結果
- test:run: {実行結果 or 未実施理由}
- lint: {実行結果 or 未実施理由}
- build: {実行結果 or 未実施理由}
- 手動スモークテスト: {実施内容と結果}

### ドキュメント更新
- {更新先}: {追記内容の要約}
```
